CAUSALONTOLOGY 4.0.0 RELEASE PLAN  (v1)

Step-by-step instructions for releasing Causalontology 4.0.0 - the attitude
kind, the predicted_occurrence kind, and the prediction_error kind (the
nineteenth, twentieth, and twenty-first object kinds; the token tier grown from
four to seven), semantics Rules 24 and 25, and the widened assertion
about-reference - to every target language's package ecosystem.

This is a versioned document (the Archive Rule applies on future bumps: a new
version is a new file and the superseded version is moved into archive/ with
git mv). Its line-1 internal header names the document and version and is never
edited on later bumps.

This plan SUBSUMES AND SUPERSEDES the Causalontology 3.0.0 release plan, which
moves to archive/ under its own Archive Rule in the same change that adds this
file. Part One states why: eighteen of the nineteen bindings never implemented
the 3.0.0 delta, so for them the 4.0.0 work folds the 3.0.0 delta in first, and
no separate 3.0.0 release remains planned.

Author: D. R. Dison / ai-university-aiu.

Acronyms are spelled out in full on first use, with the acronym in parentheses,
and the bare acronym thereafter. External proper names are kept verbatim
(SWI-Prolog, Ed25519, Secure Hash Algorithm 256-bit (SHA-256), Git, PyPI, npm,
crates.io, Maven Central, NuGet, RubyGems, Packagist, Hex, LuaRocks, Swift
Package Manager, pub.dev, Go, Zig, Hackage, CPAN, CRAN, vcpkg, Conan, the Julia
General registry).


PART ONE - THE COMMON PRE-RELEASE SECTION (applies to every language)

The single source of truth for 4.0.0 is the frozen schema and conformance suite:
the twenty-one JavaScript Object Notation (JSON) schemas under spec/schema/ and
the one hundred and thirty-seven conformance vectors under conformance/vectors/
(V01-V137). The Python reference binding passes all one hundred and
thirty-seven.

A binding MAY NOT publish 4.0.0 until it passes the 4.0.0 conformance vectors in
its own language. For every binding, in order:

 1. FREEZE. Treat spec/schema/ and conformance/vectors/ as read-only. Do not
    alter a vector to make a binding pass; fix the binding.

 2. IMPLEMENT THE 4.0.0 DELTA. Add, in the binding's own code, the three new
    token-tier kinds, the two new semantic rules, and the widened
    about-reference:
    - the attitude kind (change-order Section B): its schema and its
      identity-bearing fields (holder, attitude_type, content). The holder is a
      MODELED agent inside the ontology (a continuant or token_individual
      identifier), never a cryptographic key; the attitude_type comes from the
      CLOSED enumeration believes, desires, intends, knows, expects, fears
      (extensible only by a future major version); the content is a reference by
      identity to ANY content object INCLUDING another attitude, which is what
      makes nesting (A believes that B believes X) expressible. An attitude
      records what the holder's mind CONTAINS, not what is true - the content
      may be false - and it carries no strength of its own; it is asserted,
      graded, and retracted through the ordinary provenance layer.
    - the predicted_occurrence kind (change-order Section A): its schema and its
      identity-bearing fields (instantiates, interval, predictor, and strength
      when present). It records an occurrence a predictor EXPECTS: instantiates
      names the occurrent type predicted; the interval carries exactly ONE
      temporal dimension (see Rule 24 below); the predictor is a modeled agent
      or construct, not a key; strength is OPTIONAL and identity-bearing when
      present - the predicted probability, part of the predicted content itself,
      a bare number deliberately without the 0..1 bounds of assertion strength.
      A prediction is NOT an assertion that the thing happened: a forecast and a
      report of comparable content mint DIFFERENT identifiers.
    - the prediction_error kind (change-order Section A): its schema and its
      identity-bearing fields (predicted, observed, discrepancy). It pairs a
      predicted_occurrence with the token_occurrence that did or did not fulfil
      it: observed is OPTIONAL, and its absence means the prediction was
      fulfilled by nothing; discrepancy is the signed, graded number read
      actual-minus-expected (positive exceeded expectation, negative fell short,
      zero fulfilled).
    - semantics Rule 24 (A PREDICTION IS NOT A REPORT): a predicted_occurrence
      satisfies no rule that requires a token_occurrence; its interval MUST
      carry exactly one temporal dimension - a wall-clock start (with optional
      end) or an ordinal start_tick (with optional end_tick), never both and
      never neither; a prediction_error MUST reference a predicted_occurrence,
      and when observed is present it MUST instantiate the same occurrent the
      prediction instantiates (the pairing check needs the two referenced
      OBJECTS, so bindings implement it as a helper over the objects, as the
      Python reference does with prediction_pairing_mismatch, rather than
      inside a record-local validator).
    - semantics Rule 25 (AN ATTITUDE'S CONTENT IS QUARANTINED): no reasoning,
      conflict, reachability, or classification rule may treat an attitude's
      content as asserted; content that contradicts the actual record raises no
      conflict (that mismatch IS a false belief, first-class and shareable);
      the holder is forever distinct from the signing source.
    - the widened assertion about-reference: assertion.about now accepts
      attitude, predicted_occurrence, and prediction_error identifiers - and
      cross_stratal_seam identifiers, repairing a 3.0.0 drift (the seam scheme
      had been genuinely absent from the about pattern since 3.0.0).
    - the new conformance vectors V120-V137: group X, prediction and prediction
      error (V120-V127); group Y, attitude and theory of mind (V128-V135);
      group Z, the 4.0.0 identity witnesses (V136 re-pins two frozen 3.0.0
      identifiers byte-for-byte; V137 rejects abbreviated schemes for the new
      kinds, mirroring V107).
    All of it is ADDITIVE and IDENTITY-PRESERVING: content addressing hashes
    only the fields present, so every 3.0.0 record valid under 4.0.0 keeps its
    identifier byte-for-byte (witness V136), and the binding's existing
    identity tests must continue to pass unchanged.

    THE FOLDED-IN 3.0.0 DELTA, STATED HONESTLY. Eighteen of the nineteen
    bindings never implemented the 3.0.0 delta; only the Python reference did.
    For those eighteen, step 2 therefore folds in the 3.0.0 delta FIRST, then
    lays the 4.0.0 delta on top. The 3.0.0 delta, restated compactly from the
    superseded plan:
    - the ordinal tick temporal unit: accept "ticks" as a temporal-window unit;
      order a tick window by integer comparison of tick counts; treat a tick
      window and a wall-clock window as DISJOINT dimensions and refuse to
      convert a tick count to seconds;
    - the cross_stratal_seam kind: its schema, its identity-bearing fields
      (source, target, mechanism_status, chain), and Algorithm F (non-adjacency,
      the drawn-chain rules, the coarsest-stratum home rule, and the
      contradictory-seam and contradictory absent-with-chain checks);
    - the conduit realized_by reference: an optional, identity-bearing,
      scheme-qualified reference field; unbound remains legal.
    A binding at the 2.0.0 level goes straight to 4.0.0; no binding publishes a
    3.0.0. This is why this plan subsumes and supersedes the 3.0.0 release
    plan.

 3. CONFORM. Run the binding's own port of the conformance suite until it
    reports 137/137. Green is the gate; a binding that is not green is PENDING,
    not published.

 4. BUMP THE VERSION to 4.0.0 in the binding's package manifest.

 5. TAG THE RELEASE in Git (v4.0.0, or the language-specific tag noted below;
    Go's is bindings/go/v4/v4.0.0).

 6. PUBLISH, then run the post-publish verification: install the PUBLISHED
    4.0.0 package fresh, from the registry, into a clean environment, and run
    its conformance vectors against it. Only a fresh-install green run confirms
    the release.

Signing and provenance are unchanged from 2.0.0: Maven Central artifacts are
GNU Privacy Guard (GPG) signed; see SECURITY.md.

MATURITY NOTE. The bindings are at different maturity, and this plan says so
plainly. It gives the steps and states each language's PREREQUISITE - a binding
must first implement the folded-in 3.0.0 delta where it is missing, then the
4.0.0 delta, and pass 137/137. It does not claim a language is released until
it is. Where a 2.0.0 binding is already awaiting a human step (the Contributor
License Agreement on the vcpkg and Conan ports for C++, the web-form submission
for the R package on CRAN, the PAUSE upload for Perl, the contested Julia
General registration, the Hackage account and token for Haskell), the 4.0.0
release for that language inherits the same human gate and is PENDING until it
clears - recorded here, not hidden.
